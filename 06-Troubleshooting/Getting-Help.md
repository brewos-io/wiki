# Getting Help

Community resources and support channels for BrewOS.

## Community Resources

### GitHub Discussions

- **Location**: [GitHub Discussions](https://github.com/brewos-io/firmware/discussions)
- **Purpose**: General questions, feature requests, community support
- **Best for**: Non-urgent questions, sharing experiences

### GitHub Issues

- **Location**: [GitHub Issues](https://github.com/brewos-io/firmware/issues)
- **Purpose**: Bug reports, feature requests, technical issues
- **Best for**: Bugs, technical problems, feature requests

### Documentation

- **Wiki**: This wiki documentation
- **Firmware Docs**: [Firmware Documentation](../../firmware/docs/)
- **App Docs**: [App Documentation](../../app/docs/)
- **Best for**: Learning, reference, troubleshooting

## Getting Support

### Before Asking for Help

1. **Check documentation**:
   - Review relevant wiki pages
   - Check troubleshooting guides
   - Search existing issues/discussions

2. **Gather information**:
   - Note exact symptoms
   - Check error codes
   - Review diagnostic logs
   - Document what you've tried

3. **Prepare details**:
   - Machine type and model
   - Firmware versions
   - Configuration details
   - Error messages/logs

### Asking for Help

When asking for help, include:

- **Problem description**: Clear description of issue
- **Steps to reproduce**: How to reproduce the problem
- **Expected behavior**: What should happen
- **Actual behavior**: What actually happens
- **System information**: Machine type, firmware versions
- **Error messages**: Any error codes or messages
- **What you've tried**: Solutions you've attempted

### Where to Ask

- **General questions**: GitHub Discussions
- **Bugs**: GitHub Issues
- **Feature requests**: GitHub Issues
- **Urgent safety issues**: GitHub Issues (label as critical)

## Contributing

### Reporting Bugs

1. **Check existing issues**: Search for similar issues
2. **Create new issue**: If not found, create new issue
3. **Provide details**: Include all relevant information
4. **Follow up**: Respond to questions and updates

### Contributing Code

- **Read contributing guide**: See [CONTRIBUTING.md](../../firmware/CONTRIBUTING.md)
- **Follow code style**: Match existing code style
- **Test thoroughly**: Test your changes
- **Submit pull request**: Follow PR process

### Improving Documentation

- **Wiki improvements**: Edit wiki pages
- **Report errors**: Report documentation issues
- **Suggest improvements**: Propose documentation changes

## Safety Issues

### Critical Safety Issues

For critical safety issues:
- **Immediate**: Create GitHub Issue labeled "critical"
- **Details**: Provide full details
- **Urgency**: Mark as high priority
- **Follow up**: Monitor for response

### Safety Reporting

When reporting safety issues:
- **Be specific**: Exact symptoms and conditions
- **Include logs**: All relevant error logs
- **Document steps**: How to reproduce
- **Safety impact**: Explain safety implications

## Best Practices

### Community Etiquette

- **Be respectful**: Treat others with respect
- **Be patient**: Allow time for responses
- **Be helpful**: Help others when you can
- **Be constructive**: Provide constructive feedback

### Information Sharing

- **Share solutions**: If you solve a problem, share the solution
- **Document fixes**: Document what worked
- **Help others**: Answer questions when you can
- **Contribute**: Contribute improvements
